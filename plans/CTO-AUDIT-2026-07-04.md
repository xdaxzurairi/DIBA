# CTO Audit — DIBA Skills & Architecture
*Date: 2026-07-04 · Scope: all 36 plugin skills, 41 Feature systems, hooks, memory core*

---

## Executive Summary

DIBA has strong bones: a unified memory core, a disciplined persona spec, and 36 auto-triggered skills covering memory, projects, code quality, and creativity. But it is **not yet a real assistant** — it is a library of behaviors that only activates if Zuex types the magic word and the model happens to read the right file.

Three structural problems block the "real assistant" goal:

1. **No kernel.** There is no `CLAUDE.md` at repo root. DIBA's identity, memory protocol, and command router only load if the session starts with "DIBA" *and* the model chooses to read `master-memory.md`. Every session that skips this is a session where DIBA doesn't exist.
2. **Broken automation spine.** The hooks — the only truly *proactive* layer — are hardwired to three different machines: `settings.json` → `C:\Users\Administrator\xdibax\DIBA`, `session-start.sh` → `/home/user/DIBA`, `ask-nemotron` → `C:/Users/BSM/XDIBAX`. On any machine, at most one of these works. `session-start.sh` is not wired into `settings.json` at all, so skill auto-install never runs.
3. **No forward-looking layer.** Session-briefing looks *backward* (recap). Reminders are a flat list. Routines (`main/routines.md`) exist but nothing consumes them. Nothing answers "what should Abam do today?" — the core job of a real assistant.

**Verdict: the skill portfolio is 80% healthy. The integration layer is the weak point. Fix the spine, add the chief-of-staff layer, stop adding skills.**

## Systemic Findings

### F1 — Missing kernel (CRITICAL)
No `CLAUDE.md` at root. Claude Code auto-loads `CLAUDE.md` in every session; DIBA relies instead on a manual "DIBA" incantation and a 247-line `master-memory.md` that still contains template residue (`[AI_NAME]` placeholder in its footer). **Fix (this PR):** kernel `CLAUDE.md` — identity, memory protocol, command router — auto-loaded every session, no magic word needed.

### F2 — Hook spine broken (CRITICAL)
- `settings.json` PostToolUse → absolute path on machine A (Administrator). Fails everywhere else, silently.
- `session-start.sh` (skill auto-installer) → absolute path on machine B (`/home/user/DIBA`), **and is not referenced by `settings.json`**, so it never runs anywhere.
- `ask-nemotron` → absolute path on machine C (BSM).
**Fix (this PR):** `$CLAUDE_PROJECT_DIR`-based `settings.json`, self-locating `session-start.sh`, bash port of auto-commit, hooks README documenting the Windows/PowerShell variant.

### F3 — Dual source of truth for skills (HIGH)
Skills exist in both `Feature/*/SKILL.md` and `plugins/diba-skills/skills/*/SKILL.md`; the installer layers plugin over Feature. Drift is inevitable (already happening — Feature INDEX lists systems with no plugin counterpart and vice versa). **Direction (Phase 2):** plugin = only executable source; `Feature/` = documentation + install history only. Strip `SKILL.md` from Feature dirs or mark them `status: superseded`.

### F4 — Trigger sprawl (MEDIUM)
36 skills, most marked "MUST use" / "auto-trigger", many overlapping phrases ("execute plan" ×2, "recall" ×3, "save" ×3). Every overlapping description costs selection accuracy and context. **Direction:** one owner per trigger phrase; enforce via forge-skill review checklist (added to blueprint).

### F5 — Doc drift (MEDIUM)
`master-memory.md` says "14 features" and carries `[AI_NAME]` template residue; INDEX lists 41 feature systems; skill counts drift with every merge. **Fix (this PR):** corrected counts, residue removed. **Direction:** counts live in one place only (INDEX), everything else links.

### F6 — No proactive/forward layer (CRITICAL for "real assistant")
Nothing unifies reminders + projects + routines + decisions into "here's your day." **Fix (this PR):** new `chief-of-staff` skill — morning brief, agenda, EOD wrap, weekly review. **Direction (Phase 3):** scheduled triggers (Claude Code web/routines) so the morning brief arrives without being asked.
