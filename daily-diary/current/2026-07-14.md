## 2026-07-14 (Pagi - 10:30) - CR eWorks: Notifikasi Emel Lantikan Penyelia Seksyen PROJEK

### Session summary
Implement CR baru dalam sistem eWorks — notifikasi emel automatik kepada penyelia yang dilantik bagi aduan seksyen PROJEK di kampus Shah Alam (B01). Emel dihantar secara real-time apabila penyelia berjaya disimpan dalam `lantikan_penyelia.php`. Sesi melibatkan beberapa iterasi debug untuk menyelesaikan isu data tidak lengkap dalam emel.

### Key decisions
- Trigger emel terus dalam `lantikan_penyelia.php` (real-time) — bukan Task Scheduler seperti notifikasi AC/Air
- Guna FETCH_NUM (positional index) bukan FETCH_ASSOC — ADODB+MSSQL via `dbconnection.php` tidak konsisten dalam case nama kolum
- Nama pengadu guna `em.em_id` bukan `first_name`/`last_name` — mengikut corak `cetak.php`
- `type_req=5` = INISIATIF (tidak didokumen dalam kod lama, dijumpai dalam `cetak.php`)
- Filter emel: hanya `section='PROJEK'` DAN `site_id='B01'` — kampus lain tidak terjejas

### Bug sequence (debug trail)
1. **Bug 1:** FETCH_ASSOC — `type_req` dan `em_number` kosong sebab MSSQL memulangkan key uppercase (`TYPE_REQ`) vs lowercase (`type_wr`). Fix: tukar ke FETCH_NUM.
2. **Bug 2:** `type_req='5'` (INISIATIF) tidak ada dalam `$kategoriLabel` — cuma ada 1,2,3. Fix: tambah `'5'=>'INISIATIF'`.
3. **Bug 3:** Kondisi nama hanya semak `type_req==='1'`, bukan `'5'`. Fix: tambah `|| $type_req === '5'`.
4. **Bug 4:** Fallback `?: $em_no` dalam blok `if` — bila query EOF, `$nama_pengadu` kekal `-`. Fix: pisahkan ke `if/else` block.
5. **Bug 5 (punca sebenar):** Query guna `first_name`/`last_name` tapi kolum `em` yang betul ialah `em_id`. Dijumpai dari `cetak.php` line 422. Fix: tukar query ke `SELECT em_id FROM em WHERE em_number = ?`.
